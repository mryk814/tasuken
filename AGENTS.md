## Project structure

### ディレクトリ構成の原則

プロジェクトの種類に応じて構成を選ぶ。どれを使うかは最初に決め、途中で変えない。

**単一アプリ（ほとんどの場合これ）:**

```
{project}/
├── src/
│   ├── main.jsx          # エントリポイント（描画のみ、ロジックなし）
│   ├── App.jsx           # ルートコンポーネント
│   └── styles.css        # スタイル（tokens.css を @import）
├── index.html
├── vite.config.mjs
├── package.json
└── AGENTS.md
```

**複数アプリ（モノレポ）:**

```
{project}/
├── apps/
│   └── {app-name}/       # 各アプリ（単一アプリと同じ内部構造）
├── shared/               # 共有コード（あれば）
├── package.json          # ルート（workspaces 設定）
└── AGENTS.md
```

- モノレポでは npm/pnpm workspaces を使い、依存を共有する。`file:` symlink で他アプリの `node_modules` を参照しない。
- 1アプリしか作らないのに `apps/` を掘らない。最初は単一アプリで始め、2つ目が必要になったらモノレポに移行する。

### ファイル分割の基準

- **小規模（〜500行程度）は `App.jsx` 1ファイルでよい。** 画面が増えたり、コンポーネントを3箇所以上で再利用する場合に分割する。
- **分割時のディレクトリ:**
  - `src/components/` — 再利用コンポーネント（`StatusBadge.jsx`）
  - `src/pages/` または `src/views/` — 画面単位のコンポーネント
  - `src/utils/` — ロジック・ヘルパー（`formatDate.js`）
  - `src/hooks/` — カスタム hooks（`useLocalStorage.js`）
- **CSS は `src/styles.css` 1ファイルを基本とする。** 800行を超えたらコンポーネント単位で分割してよい。
- **データ・定数**はコンポーネントファイルの先頭か、量が多ければ `src/data/` に置く。
- 使わないディレクトリを先に作らない。必要になった時点で作る。

### 命名規則

| 対象 | 規則 | 例 |
|------|------|------|
| プロジェクト / アプリ名 | kebab-case | `run-workbench`, `book-manager` |
| コンポーネントファイル | PascalCase.jsx | `StatusBadge.jsx`, `RunList.jsx` |
| ユーティリティ・hooks | camelCase.js | `useLocalStorage.js`, `formatDate.js` |
| CSS class | kebab-case（flat） | `run-row`, `panel-heading`, `badge-completed` |
| localStorage キー | `{app-name}:{キー名}` | `run-workbench:runs`, `run-workbench:theme` |

## Application architecture principles

技術やディレクトリを先に決めるのではなく、利用者の作業、正式データ、失敗時の復旧方法から
アプリの構造を決める。構造の良さはファイル数ではなく、変更箇所を予測でき、データを失わず、
実行経路を端から端まで検証できるかで判断する。

### 1. まず作業の一周を定義する

- 画面やテーブルを作る前に、利用者が何を受け取り、どう整理し、何を判断し、どこへ持ち出すかを書く。
- 機能は入口、一覧、詳細、変更、保存、再表示、削除または復元までを一つの縦断経路として設計する。
- データモデルの項目数ではなく、利用者の目的と判断単位で画面・機能境界を決める。
- 新機能を独立画面として増やす前に、既存の作業経路へ自然に統合できないか確認する。

### 2. 正式データの正本を一つにする

- 同じ事実を複数のstate、キャッシュ、ファイル、DBへ別々の規則で保存しない。
- 正式データ、派生表示、一時入力、UI状態を区別する。派生値は正本から計算し、一時入力は保存成功まで残す。
- 一覧、詳細、編集、Exportは同じ正本の異なる投影として扱い、画面ごとの独自コピーを作らない。
- 保存APIの成功後に正式stateを更新する。失敗時は既存データと入力値を壊さない。

### 3. 権限と副作用の境界を明確にする

- UI、状態管理、業務処理、永続化、OS・外部サービス操作を混ぜない。
- ファイル、DB、ネットワーク、クリップボードなどの副作用は、許可された少数の境界からだけ呼ぶ。
- 境界では入力を検証し、内部では型付けされた業務語彙を使う。汎用的な「何でも実行できるAPI」を公開しない。
- 複数データを同時に変える処理は、途中状態を見せないtransactionまたは同等の仕組みで完結させる。
- entry pointは接続と起動に限定し、業務ロジックを置かない。

### 4. 状態は必要な範囲にだけ置く

- 複数画面で共有し、保存対象になる正式データだけをglobal storeへ置く。
- フォーム途中値、開閉、ページ固有filterなど、閉じれば捨ててよい状態はcomponentや画面内に残す。
- Storeは「何でも入れる箱」にせず、データ操作、UI状態、ページ固有状態の責任を分ける。
- 派生値はselectorまたは純粋関数にし、同じ集計ロジックを画面ごとに複製しない。

### 5. 小さく始め、責任が見えた時点で分ける

- 将来使うかもしれない層、interface、ディレクトリを先回りして作らない。
- 小規模なうちは一つのファイルでもよいが、画面・業務処理・副作用の責任が混ざり始めたら分割する。
- 分割は行数だけで判断しない。「変更理由が異なる」「単独で検証したい」「3箇所以上で再利用する」を主な基準にする。
- abstractionは重複を隠すためではなく、意味のある契約や変更境界を一箇所にするために作る。

### 6. 新機能は縦に接続する

- UIだけ、保存だけ、未接続componentだけを先に量産しない。
- 型・データ契約 → 永続化 → 業務処理 → API境界 → state → UI → 実動検証の順で一つずつ通す。
- 各段階でpackage可能・起動可能な状態へ戻し、大規模変更でも壊れた中間状態を長く残さない。
- 新しい経路が成立したら旧経路を撤去し、同じ責任を持つ実装を二重に残さない。

### 7. 移行は互換条件から設計する

- 大きな移行の前に、既存データ、ファイル形式、API、主要操作、build/package結果を基準として固定する。
- データschema、Import/Export形式、保存場所を不要に変えない。変更が必要ならversionと移行手順を用意する。
- UIから機能を消すことと、既存データを読めなくすることを分ける。
- 移行は実行基盤、契約、永続化、state、UIのように段階化し、各段階で互換性を検証する。
- 移行完了後は旧entry、旧adapter、旧成果物を撤去し、正本がどれか迷わない状態にする。

### 8. 失敗と復旧を通常経路として設計する

- 読込、空、エラー、成功だけでなく、再試行、Undo、Import差分、backup、再起動後の復元を設計に含める。
- エラーは握りつぶさず、原因と利用者が次に行う操作を示す。
- 外部入力はプレビューと検証を挟み、既存データを直接上書きしない。
- ローカルファーストでは保存場所、backup方法、別端末への移行方法を明示する。
- 削除や一括変更は、影響範囲に応じてUndo、確認、transaction、Snapshotを使い分ける。

### 9. 完成は実行境界を越えて確認する

- componentが描画されたことやbuild成功だけを完成条件にしない。
- 作成 → 保存 → 再読み込み → 再表示 → 編集 → 削除・復元を、正本データを通して確認する。
- Webはproduction build、デスクトップはpackage版、データツールは実データまたは代表fixtureで確認する。
- READMEの起動・build・packageコマンドを、記載どおりの環境で実行して成果物まで確認する。
- 自動テストで置き換えられないOS、ファイル、DB、配布物の境界は、実動smokeを残す。

### 10. 文書も構造の一部として保つ

- 実装変更時はREADME、構造図、保存場所、検証コマンドも同時に更新する。
- 文書には理想構成だけでなく、現在の正本、保留事項、互換条件を書く。
- 設計判断は特定ライブラリ名だけで説明せず、守りたい責任・データ・失敗条件を記録する。
- 実装と食い違った古い構造図、旧entry、完了済み移行TODOを残さない。

## Tech stack

### 既定の選択

新しいプロジェクトを始めるとき、特に理由がなければ以下を使う。用途に応じてプラットフォームを選ぶ。

**Web アプリ（既定）:**

| 項目 | 選択 | 備考 |
|------|------|------|
| 言語 | **JavaScript**（JSX） | TypeScript は複雑さに見合う規模になったら検討（要相談） |
| ビルド | **Vite** | 設定は `vite.config.mjs` |
| UI | **React** | `.jsx` 拡張子 |
| CSS | **vanilla CSS + design-standard/tokens.css** | CSS Modules・CSS-in-JS は使わない |
| アイコン | **Tabler Icons** (`@tabler/icons-react`) | プロジェクト内で他のセットと混ぜない |

**デスクトップアプリ:**

| 項目 | 選択 | 備考 |
|------|------|------|
| フレームワーク | **Electron + electron-vite** | Main / Preload / Renderer を分離し、開発・ビルド経路を統一する |
| 言語 | **TypeScript** (`strict: true`) | Main / Preload / Renderer / shared 契約を型付けする |
| UI | **React + Tailwind CSS** | `design-standard/tokens.css` を見た目の正本とし、複雑な描画は通常CSSを併用する |
| 状態管理 | **Zustand** | 正式データ・共有非同期状態に使い、ページ内だけの状態は `useState` に残す |
| アイコン | **Tabler Icons** | アプリ内で他セットと混ぜない |
| トークン | Web と同じ `tokens.css` を使う | |
| DB | **SQLite + better-sqlite3** | Main Process の Repository だけがDBへアクセスする |
| 配布 | **electron-builder** | WindowsはNSIS installerとportableを標準成果物にする |
| ファイル操作等 | Electron の Node.js API / typed IPC | RendererからNode.js、Electron、SQLiteを直接触らない |

- 個人用Electronアプリの詳細な既定作法は [`docs/desktop-app-standard.md`](./docs/desktop-app-standard.md) を参照する。
- 実行経路は `React → Zustand → window.api → Preload → IPC → Service → Repository → SQLite/OS` を基本とする。
- PDF.js はPDF閲覧が製品機能に含まれる場合、electron-updater は更新配布先と署名方針が決まった場合に追加する。

**データ分析・可視化（Python）:**

| 項目 | 選択 | 備考 |
|------|------|------|
| 言語 | **Python** | |
| 可視化 | **Plotly**（インタラクティブ） | matplotlib は静的な出力が必要な場合のみ。基本は Plotly を優先 |
| 簡易アプリ化 | **Streamlit** または **Dash** | 結果を触って確認するための軽い画面。本格的な Web アプリにはしない |
| データ操作 | **pandas** / **polars** | プロジェクトに合う方を選ぶ |
| ノートブック | Jupyter は探索段階で使ってよい | 成果物として残す場合はスクリプト（`.py`）に整理する |

- Plotly のカラーパレットは `tokens.json` のチャート色（`chart-1`〜`chart-6`）を `plotly.io.templates` やレイアウトの `colorway` に設定して合わせる。
- 凡例・軸・背景の配色もトークンに寄せる。完全一致は不要だが、burgundy をアクセントに使い、背景はクリーム系にする方向性は保つ。
- 色だけに頼らず、線種・マーカー形状を併用する原則はデザインガイドと同じ。
- Streamlit/Dash で画面を作る場合もデザインガイドのUX規約（4状態、エラー文、操作フィードバック等）は意識する。ただし CSS レベルのトークン厳守は求めない。

**CLI / スクリプト:**

| 項目 | 選択 | 備考 |
|------|------|------|
| 言語 | **Python** または **Node.js**（ESM） | データ処理寄りなら Python、Web ツール寄りなら Node.js |
| 出力の色 | `tokens.json` のカラーを参考にしてよいが、ターミナル標準色で十分 | |

- プラットフォームの選択は自由にしてよい。迷ったら Web アプリから始める（ブラウザさえあれば動く）。
- デザイントークンは全プラットフォーム共通。CSS が使えない環境では `tokens.json` の値をプラットフォームの色・寸法定義に移し替える。

### 技術選定の判断基準

- **状態管理**: Webアプリは`useState` + `useReducer`を基本とする。複数画面を持つElectronアプリはZustandを既定とし、正式データ・共有非同期状態だけをStoreへ置く。
- **永続化**: ブラウザ完結なら `localStorage`。デスクトップアプリはローカルファイル（JSON / SQLite）。複数端末同期やデータ量が大きい場合はサーバー/DB（要相談）。

### やらないこと

- 使う予定のないツール・設定を「念のため」入れない（ESLint, Prettier, husky 等は必要になったら足す）。
- ボイラープレートジェネレータ（create-react-app, create-next-app 等）の出力をそのまま使わない。不要なファイルが大量に入る。必要なものだけ手で置く。

## データモデルと永続化の標準

プラットフォームに依存しないデータ設計の規約。SQLite・JSONファイル・localStorageのいずれでも従う。

1. エンティティは共通カラムを持つ: `id`（UUID。連番IDは使わない）／ `created_at` ／ `updated_at` ／ `deleted_at` ／ `version`（保存するたびに+1する。取り込み時に新旧を判定する基準にする）／ `source`（"manual" | "imported" | "quick-capture" 等、データの由来を示す文字列）。
2. 日時は ISO 8601 形式で保存する。日付のみのフィールドは `YYYY-MM-DD` 形式にする。ローカル日付文字列を `toISOString().slice(0, 10)` で作らない（UTCとのずれで日付がずれる）。`getFullYear` / `getMonth` / `getDate` から組み立てる共通ヘルパー関数を1つ用意し、日付文字列の生成箇所すべてでそれを使う。
3. 削除は論理削除（`deleted_at` にタイムスタンプを入れる）を既定にする。物理削除は明示的な要件がある場合だけ行う。削除は、完了・受領・却下・アーカイブなどの状態変更とは別の操作として設計する（同じボタン・同じ関数で兼用しない）。
4. 関連エンティティが削除された場合の扱いは、型ごとに列挙した delete policy 関数1箇所にまとめる。参照は nullify（切り離す前の参照値を退避フィールドへ記録する）、子は cascade（削除元を示すマーカー付きで論理削除する）のいずれかにする。restore 操作は、退避した参照値とマーカーから元の関連を完全に復元できることを条件にする。新規エンティティを追加するときは、この delete policy 関数への追記を必須の作業項目にする。
5. enum に相当する値は文字列 union（`as const`）で定義する。DBに保存する値は英語 snake_case にし、画面に表示するラベルは対応表1ファイル（例: `labels.ts`）に集約する。UIコード内に表示文言を直書きしない。enum フィールドを1つ追加したら、同じ変更の中でラベル対応表にも追記する（内部コードがそのまま画面に表示される事故を防ぐ）。
6. ユーザーが並べ替える対象にだけ `sort_order`（数値）カラムを持たせる。配列に格納されている順序に暗黙で依存しない。
7. migrationは「スキーマのバージョン番号」と「連番のmigrations配列」で管理する。適用済みのmigrationは内容を書き換えない。DBのバージョン番号がアプリのバージョン番号より新しい場合はアプリの起動を拒否する。
8. 参照整合性とデータの循環参照は、DBのFK制約に任せず、保存時にアプリ層で検証する。存在チェック（参照先が存在するか）と循環チェック（親子・依存グラフの中で閉路が生じないか）の両方を行う。
9. DBのrow型とアプリのDomain型は別の型として定義し、永続化層（Repository相当）で変換する。
10. 「作成→一覧→詳細→編集・削除→永続化」という利用者が到達できる一連の操作（成立ループ）が通らないテーブル・カラムを先に作らない。「将来使うかもしれない」構想はバックログに文章で書き、schemaには反映しない。
11. データモデルの世代交代では、移行期間中は旧形式と新形式の両方を読めるようにする。旧形式を参照するフィールド・テーブルは `legacy_*` という命名で識別できるようにする。
12. Import やSnapshot取り込みは必ず次の順で行う: 検証 → プレビュー（既存データとのdiffを新規・同一・更新・衝突に分類して提示） → ユーザーが採用/上書き/無視を決定 → 適用。検証やプレビューを経ない無条件の直接上書きをしない。
13. 一覧のフィルタ条件が複数の独立したフィールド（例: 状態×種別）の組み合わせで決まる場合、片方のフィールドだけを変更した結果「保存されてはいるが、どの一覧にも表示されない」データが生まれないかを設計時に確認する。
14. 変更履歴（何が・いつ・なぜ変わったか）は後付けの監査ログとしてではなく、意思決定を再現するために必要な箇所へ最初から設計に含める（原則8「失敗と復旧を通常経路として設計する」と整合させる）。
15. **自動バックアップ**: ローカルファーストのアプリは、起動時または終了時に自動でバックアップ（Snapshot）を書き出し、N世代（既定5）をローテーション保持する。手動Exportだけに頼らない。保存先と世代数は設定画面に出す。
16. **書き込み経路の一本化**: 複数のウィンドウ・プロセスから同じデータを扱う場合、書き込みは単一の経路（例: Main Process の Repository）に集約し、変更通知で他のウィンドウへ反映する。各ウィンドウが直接ファイル・DBへ書かない。
17. **Exportの往復性**: ExportしたデータをImportし直したとき同じ状態に戻ることを、Export/Import機能の検証項目に含める。

## AI連携の標準

アプリとLLMを連携させる場合の既定方式。

1. LLM APIをアプリに内蔵しない構成を既定にする。標準の連携方式は**クリップボード往復**にする: プロンプト文字列を生成する → クリップボードへコピーする → 利用者が外部AIへ貼り付ける → 返答（JSON等）をアプリへ貼り戻す → パースする → プレビューを表示し採用/無視を選択させる → 保存する。
2. MCP等でAIにアプリへ直接アクセスさせる場合は read-only を既定にする。書き込みを許可する場合は「提案（proposal）エンティティを作成するだけ」の操作に限定する。提案を実データへ反映するのは、利用者がUI上で内容を確認し採用を選んだ後にする。
3. AIからの取り込みも、上記「データモデルと永続化の標準」の Import と同じ「検証→プレビュー→決定→適用」の手順に従う。取り込みが失敗した場合、既存データと入力途中の値の両方を保持したままにする。

## セキュリティと秘密情報

1. 秘密情報（APIキー・トークン・パスワード）をソースコード・設定ファイルに直書きしない。リポジトリにコミットしない。
2. 秘密情報は環境変数または `.env`（必ず `.gitignore` に登録）へ置く。リポジトリには値を空にした `.env.example`（キー一覧）を置く。
3. Electronで利用者の資格情報を保存する場合は `safeStorage` で暗号化する。平文のJSON・localStorageに書かない。
4. ログ・エラーメッセージ・Export・クリップボードコピーに秘密情報を含めない。
5. テレメトリ・クラッシュレポート等の外部送信は既定で入れない。

---

## エラーハンドリングとログ

1. エラーを握りつぶさない。`catch` して何もしない場合は、無視してよい理由をコメントで書く（例: 効果音の再生失敗は機能に影響しない）。
2. 想定内の失敗（検証エラー・ユーザーによるキャンセル）と想定外の失敗（例外）を区別し、想定外だけを error として扱う。
3. UIに出すエラーは原因＋直し方（design-guide §7）。ログに残すエラーは発生箇所・原因・スタックを含める。両者は役割が違うので同じ文字列を使い回さない。
4. デスクトップアプリは Main 側で userData 配下にファイルログを書く。レベルは info / warn / error の3段にする。
5. ログに秘密情報とデータ本文を書かない。エンティティはIDと件数で書く。

- 既存機能の変更がある場合には基本的にフォールバックを作らずに完全に置換することを目指す。
- フォーマットは一般的な整形に従う（セミコロンあり、引用符はプロジェクト内で統一）。
- `import` はフレームワーク → 外部ライブラリ → ローカル の順。

## Design standard

このプロジェクトには確定済みの個人デザイン標準がある。
UI・GUI・グラフ・ツールの見た目を作る作業では**必ず従うこと**。

**詳細は以下を参照（これが唯一の真実）:**

- 原則・挙動・アンチパターン・文章規定: [`design-standard/design-guide.md`](./design-standard/design-guide.md)
- トークン（CSS変数、ライト/ダーク両対応）: [`design-standard/tokens.css`](./design-standard/tokens.css)
- トークン（JSON、CSS以外の環境用）: [`design-standard/tokens.json`](./design-standard/tokens.json)

### 要約（全文は design-guide.md）

- 方向性: 「親しみ柔らかめ × burgundy」
- アクセント `#8A2F3B` は操作/重要のみ。エラーの赤(`#CE3B3B`)と混同しない
- 非ベース色は「意味固定の状態色（danger等・流用禁止）」「ワークフロー状態色（--color-status-*・対応表を1つ）」「カテゴリ色（chart-*・区別のみ）」を分ける。タグ/バッジ/バーは色を消してもテキストで識別できること（色は補助）。多値の状態は --color-status-* で進行中を前に・終端を後退させ、ニュートラル一色に潰さない
- 面と背景の明度差で奥行きを出し（影に頼らない）、主役の指標を1つサイズで際立たせ、現在地は淡背景だけに頼らず標識を2要素以上で示す（平坦・無階層を避ける）
- コンパクト密度・角丸7px・丸ゴシック(Nunito系)・数値は等幅で桁揃え
- focusリング必須・ライト/ダーク両対応
- 主要なデータ領域・非同期フローで、該当する4状態（読込中・空・エラー・成功）を到達可能にする
- エラーでもフォーム入力を消さない。復元可能な操作はundo、不可逆・大量・外部影響のある操作は具体的に確認
- 操作要素はdefault/hover/activeの3状態必須。フィードバックは100ms以内
- 通常画面は説明文ゼロから始める。状態説明は省略しない。1画面の主目的は1種類、強いPrimaryは同時に1つ
- 同一階層で並列に判断させる主要操作は7個までを目安にする
- 絵文字アイコン・紫グラデ・影マシマシ・テンプレ構成の"いかにも"禁止
- コンポーネントの過剰設計を避ける
- エラー文は原因＋直し方。ボタンは動詞。トーンは簡潔・落ち着いた敬体
- テキストファイルはUTF-8で読み書きする
- データモデルを1:1で画面化しない。近い目的の情報は統合、ナビは利用頻度で設計、新画面より既存導線への統合を優先（IA）。良い整理は要素数でなく「次にどこを押すか迷う回数」を減らすこと
- 機能は「作成→一覧→詳細→編集削除→永続化」が通って初めて成立。到達不能な画面・保存だけのデータ・未接続コンポーネントを完成扱いにしない
- 入力（粗く速く）と整理（分類・日付・関連付け）を分離。未定/仮/確定/実績を同じ日付に潰さず、確度を保ったまま保存できるようにする
- 一覧=探す/整理、詳細=理解、編集=変更で責任を分け、同一データの投影として食い違わせない。同じ入力目的の導線を重複させない
- エンティティ操作は「行クリック→詳細ドロワー→編集ドロワー」の2段階を既定にする。削除は編集ドロワー末尾の Danger Zone に隔離し、実行は即時＋undoトーストにする。削除と完了・アーカイブ等の状態変更を混同しない（§18）
- 右クリックのコンテキストメニューはショートカットであり、唯一の導線にしない。メニュー内の全操作は可視のUI（ボタン・ドロワー）からも到達可能にする（§19）
- 単一選択は詳細表示、複数選択は一括操作と役割を分ける。チェックボックス列は一括操作を実装してから出す（§20）
- インライン編集・ドロワー・ページ・モーダル・トーストの使い分けは決定表で判断する（§21）
- フィルタ・タブ・表示切替の状態は保存して復元する。結果0件は空状態と区別し、フィルタ解除の導線を出す（§22）
- トーストは `showToast(message, tone)` のように tone を明示引数で渡す。tone（success/warning/danger/info）の意味は固定し、文言の一致判定で分岐しない（§23）
- カウントバッジは「行動を促す数（未処理・要確認・期限超過等）」だけに付け、総件数には付けない。ルートID変更時は旧→新のエイリアス表に追記する（§24）

## 作業の進め方

このプロジェクトでの実装作業を進める際のプロセス規定。

1. 実装に着手する前に、対象の現状（画面・データ・導線）を読み、変更計画を短く書いてから着手する。
2. 大きな変更は複数のフェーズに分割する。各フェーズは単独でビルド・起動でき、必要であればリリース可能な状態に保つ。
3. 大きな移行・リファクタリングに着手する前に、「非交渉の互換条件」（例: 既存データがそのまま読める・Export形式を変えない・CRUDの結果を変えない）を5項目前後で列挙する。すべてのフェーズの完了時にこの条件を検証する。
4. 新しい保存機能は「型・契約 → 永続化 → 業務処理 → API境界 → state → UI → 実動確認」の順に縦へ1本ずつ接続する（Application architecture principles 原則6の実行手順）。
5. リファクタリング作業の最中に新機能を混ぜない。機能追加作業の最中に無関係な改修を混ぜない。作業中に思いついた別のアイデアはバックログに書いて、今の作業を続行する。
6. 新規エンティティを追加するときのチェックリスト:
   - (a) 削除導線を実装前に1行で確定する（一覧から／詳細から／編集から／Undo の4項目それぞれの動作を決める）。
   - (b) 「データモデルと永続化の標準」の delete policy 関数へ追記する。
   - (c) enum を追加する場合はラベル対応表を同時に作る。
   - (d) 成立ループ（作成→一覧→詳細→編集削除→永続化）を実機で確認する。
7. Definition of Done は次を満たすこととする: typecheckとbuildが成功する／成立ループを実画面で確認する／関連ドキュメントを更新する／実施した検証を作業報告に含める。失敗した検証を成功として報告しない。実施していない検証を実施したと書かない。
8. 意図的に見送る依存・機能は、記載しないまま済ませず、「導入条件」とセットで記録する（例: 自動更新は配布先と署名方針が決定した後に導入する）。
9. 表示文言に対する正規表現・キーワード一致で挙動を分岐させるような暗黙の結合を作らない。分岐が必要な場合は明示的な引数・型として渡す。

### 完成前セルフチェックリスト

完了報告の直前に、次の10項目を機械的に確認する。全部YESであること。NOがあれば直してから報告する。

1. 成立ループ（作成→一覧→詳細→編集削除→永続化→再起動後の再表示）を実画面で通したか
2. 該当する4状態（読込中・空・エラー・成功）に到達できるか
3. 失敗時に入力値と既存データが残るか
4. focusリングが出るか。Tabだけで主要操作を完了できるか
5. 新しいenumにラベル対応表を作ったか（内部コードが画面に出ていないか）
6. 新しいエンティティの削除導線4項目（一覧から/詳細から/編集から/Undo）が埋まっているか
7. トーストのtoneは意味どおりか（success=成功/warning=削除+undo/danger=失敗/info=その他）
8. 色・余白・角丸・文字サイズをトークン変数から取ったか
9. 秘密情報がコード・ログ・Exportに混ざっていないか
10. typecheck / build が成功し、実施した検証を報告に書いたか

## Testing

テストの走らせ方が固まるまでの間も、検証は「機能が成立しているか」を基準にする（詳細は design-guide.md §17）。

- **「DOMに表示された」で完了にしない。** DB保存 → アプリ再読み込み → （デスクトップなら）パッケージ版起動まで確認し、保存境界を越えて検証する。
- **削除・整理のあとは残った導線を実画面で検証する。** ビルド成功では不十分。到達・作成・保存・再表示・削除まで通す。
- **失敗時に作業途中を失わないことを確認する。** 保存失敗・解析失敗・Import失敗でも、入力値と既存データが残ること。
- 開発サーバーで動くだけで完成にしない。READMEに書いた起動・ビルドコマンドが実環境でそのまま成功することを完成条件にする。

## Git conventions

- コミットは「1フェーズ＝1コミット」を基本にする。ビルドが通らない中間状態でコミットしない。
- コミットメッセージは日本語1行（50字目安）で「何を・なぜ」を書く。詳細が必要なら空行の後に本文を書く。例: `タスク削除にundoトーストを追加（確認ダイアログの置き換え）`
- lockfile更新・自動整形などの機械的な巨大diffは、手書きの変更と混ぜず独立コミットにする。
- ブランチ: 個人開発では main 直コミットでよい。実験的な大変更・移行だけブランチを切る。

## Boundaries

### Always do

- design-standard/ のトークンを参照する
- 主要なデータ領域・非同期フローで、発生しうる4状態を設計する
- focusリングを表示する
- 数値は等幅で桁揃え
- 色・余白・角丸・文字サイズ・影・動きはトークン変数から
- 日本語を含むファイルはUTF-8を明示して扱う
- Project structure / Tech stack セクションの構成・命名規則に従う
- 表・一覧にはクリップボードコピーの導線を用意する
- 機能は入口→一覧→詳細→編集削除→永続化のループが通り、通常操作から到達できることを確認する
- 一件操作だけでなく、量が増えたときの一括操作・コピー・エクスポートの逃げ道を用意する
- ローカルファーストでは保存場所・再起動後の復元・バックアップ・Import差分を明確にする（「消えない」「戻せる」を見た目より先に保証する）
- データの持ち出し・取り込み（Excel/CSV/AI/ファイル）の導線を保ち、アプリ内に閉じ込めない
- 新規エンティティは「作業の進め方」のチェックリストに従う
- ルートID変更時は旧→新エイリアス表に追記する
- enum追加時はラベル対応表を同時に作る
- ローカルファーストでは自動世代バックアップを実装する
- 完了報告の前に完成前セルフチェックリストをなぞる

### Ask first

- design-standard/ 自体の変更
- 大きなリファクタリング
- CSS-in-JS の導入
- サーバー/DB の新規導入
- Tech stack の既定以外のフレームワーク選定（Tauri、Flutter 等）

### Never do

- トークンから外れた色・余白・角丸・文字サイズを場当たり的にハードコード
- burgundy をエラー表示に使う
- 11px未満の文字
- 絵文字アイコン
- 説明文の洪水・テンプレ構成の"いかにも"
- `file:` symlink で他パッケージの `node_modules` を参照する
- ブラウザ/OS の既定キーボードショートカットを上書きする
- ボイラープレートジェネレータの出力をそのまま使う
- 未接続のコンポーネント・到達不能な画面・保存されるだけのデータを完成機能として数える
- 使わせたくない機能の初期データ（seed）を作る／見栄え用のダミーで画面を埋める
- データモデルのオブジェクトを機械的に1つずつ独立画面に割り当てる
- UIから消すこととデータを物理削除することを混同する（利用実態の確認前にデータモデルを廃止しない）
- UIから到達できないテーブル・カラムをschemaに足す
- 削除を状態変更と混同する（完了・アーカイブを削除で代用する、またはその逆）
- 表示文言に依存した分岐を書く
- UI層からDB・ファイル・OSへ直接アクセスする（境界コンポーネントを経由しない）
- カウントバッジに総件数を表示する
- 秘密情報をコード・リポジトリ・ログ・Exportに含める
- catchして理由コメントなしで握りつぶす
