# Tasken UX/IA 交通整備計画

作成日: 2026-07-02
状態: ドラフト（フェーズ順・保留事項の確定待ち）

この文書は、現状の画面・導線の棚卸し、過剰積載の診断、そして段階的な整理計画をまとめる。
IAの現状の正本はコード（`src/renderer/src/pages/routes.ts` / `components/shell.tsx` / `WorkspaceApp.tsx`）であり、本文書は「どう変えるか」の計画のみを扱う。
判断基準は `AGENTS.md` のアプリ構造原則と `design-standard/design-guide.md` に従う。

## 2. 診断

### 2.1 到達不能・行き止まり・保存だけのデータ

機能成立ループ（入口→一覧→詳細→編集削除→永続化）が閉じていないものが残っている。
`AGENTS.md` の Never do「到達不能な画面・保存されるだけのデータを完成扱いにしない」に照らすと、最優先の整理対象。

- **MilestonePage は到達不能。** `pages/MilestonePage.tsx` は存在するが `routes.ts` にも `WorkspaceApp.tsx` の pages マップにも未登録で、どこからも開けない。機能自体は Today「近いマイルストーン」と Timeline で代替されている。
- **付箋メモが行き止まり。** サブタイトルは「Inbox整理に乗せる前の走り書き」だが、`InboxPage` は `kind !== "micro_memo"` で除外し、capture_entry ドロワーにも kind を変える UI がない。付箋メモをタスク・メモ・Inboxへ昇格させる道が存在しない（編集・削除・コピーのみ）。
- **孤立したドロワーフォーム。** `drawer.tsx` と `WorkspaceApp.saveForm` には `source_record` / `field_definition` / `task_dependency` / `reference` の編集フォームと保存分岐があるが、これらの型で `openDrawer` を呼ぶ画面がひとつもない。
- **UIのない保存データ。** `saved_view` / `log_entries` / `field_values` / `entity_sources` はDB・型・Snapshot IOには存在するが、参照・作成できる画面がない。

### 2.2 Todayの過剰積載

Todayは「今日見るものを一か所に集める」ハブだが、集めすぎて主目的（今日なにをやるか決めて着手する）が埋まりつつある。

- ヘッダーに5ボタン（ミニ表示 / 更新 / コピー / 活動ログ出力 / タスク追加）。
- その直下に **Activity Log ストリップが常設**（日付入力+コピー+出力先変更）。これは報告・記録用ツールであり、今日の判断より上に置く理由がない。
- 以降、フォーカスヒーロー + メトリクス4 + セクション7（今日やること / 期限切れ / Inbox未整理 / 予定なし / 近いマイルストーン / 期限が近い待ち / 最近の現在地）。
- 各行に操作要素7個（完了 / 旗 / 今日トグル / タイトル / +1d / +7d / 行クリック）。design-guide「同一階層で並列に判断させる主要操作は7個まで」の上限に行単位で張り付いている。
- Activity Log 生成ロジック約70行（`buildActivityLog`）がページコンポーネント内に同居。

さらに同じ情報が**3面同時**に出る: サイドバーのカウント、Todayのセクション、ContextPane（期限切れ・待ち・現在地）。

### 2.3 正本の投影分裂

`AGENTS.md`「一覧・詳細・編集・Exportは同じ正本の異なる投影として扱う」に対し、投影がページ単位で割れている。

- **note の投影が4面に分裂。** Notes（スコープに prompt を含む）/ Prompts（独立ページ）/ 報告書（HomePage の報告書セクション: report / report_prompt）/ AI IO（週報レポート形式・Document Publish 対象管理）。同じ note_type の編集導線が複数画面にあり、「プロンプトを直すのはどこか」を毎回思い出す必要がある。
- **resource の投影が2面に分裂。** チャット参照（`isChatReference` = URL推定）と Notes & Resources（それ以外）。URL推定が境界なので、判定を跨いで移動する手動導線がなく、誤判定時の逃げ道がない。
- **AI往復の導線が5面に分散。** AI IO（プロンプトコピー→貼り付け→取り込み）/ Proposal Inbox（MCP write の確認）/ Notes 内 Knowledge 抽出 / Knowledge 内プロンプトコピー / チャット参照。「AIと往復して確認して取り込む」という同じ作業が、入口ごとに別画面になっている。

### 2.4 サイドバーのカウント過多

10ルートにカウントバッジが付くが、性質が混在している。

- 行動を促す数（未整理 Inbox、pending の AI提案、今日、期限つき待ち）
- ただの在庫数（Notes / Knowledge / Prompts / チャット参照 / 付箋メモの総数）

全部が同格で光ると、本当に見るべき数（Inbox未整理・AI提案）が埋もれる。

### 2.5 表現の不統一・磨き残し

- 命名: ナビ「チャット参照」vs ページ「チャット参照棚」、「Proposal Inbox」vs「AI Proposal Inbox」、日英混在（Inbox整理 / 付箋メモ / Timeline / Knowledge）。
- コピー導線の文言ゆれ:「一覧をコピー」「コピー」「URLをコピー」。
- **手動「更新」ボタン**（Today / Inbox）。`onWorkspaceChanged` で外部変更は自動反映される設計であり、ボタンが必要なら通知経路の欠陥、不要なら純粋なノイズ。
- **トーストの色がメッセージ文言の正規表現で決まる**（`WorkspaceApp.toastTone`）。文言を直すと色が壊れる暗黙結合。
- ルート名 `home` = テーマダッシュボード。コード上の紛らわしさ。
- ToDo の完了一覧が `todo-done` という擬似ルートで、ToDo 内フィルターと二重表現。

## 3. 交通整備の方針

### 3.1 守る原則

1. **1画面1主目的。** ハブ（Today / HomePage）は「要約と入口」に徹し、操作の本体は一覧画面（ToDo / Waiting / Inbox / Timeline）に置く。
2. **入口は軽く多くてよいが、整理の合流点は1つ。** 記録の入口（capture window / Alt+N / インライン追加）は維持し、「後から整理する場所」を Inbox に一本化する。
3. **投影はページ分割ではなく絞り込みで表現する。** ページを分けるのは目的（探す/整理する/理解する）が違うときだけ。
4. **到達不能・行き止まり・保存だけのデータをゼロに保つ。** 今後も画面追加時は成立ループの確認を完成条件にする。
5. **UIからの撤去とデータ削除を分ける。** 孤立フォームの撤去でも schema・Snapshot 互換・読み込みは維持する（互換条件）。

### 3.2 目標ナビゲーション（To-Be）

```
今日の運用   Today（減量） / ToDo / Waiting
横断        Inbox（付箋メモをレーン統合） / Timeline
知識整理     Knowledge / Notes（Prompts吸収） / チャット参照
テーマ別     すべてのテーマ / 各テーマ
ツール       AI連携（AI IO + Proposal Inbox） / Settings
```

- ルート数 14 → 11。区分は5つのまま、各区分内は最大3項目。
- カウントバッジは「行動を促す数」だけに限定: Inbox未整理 / AI提案 pending / 今日 / 期限切れ相当。在庫数は外す。

### 3.3 保留事項（実装前に決める）

| 論点 | 選択肢 | 推奨 |
|------|--------|------|
| Prompts の独立ページ | (a) Notesのスコープへ吸収 (b) 独立維持しNotes側のpromptスコープを撤去 | (a)。件数が少なく用途別グルーピングはNotes内フィルタで再現可能。報告書プロンプトの主導線はHomePageにあり失われない |
| Proposal Inbox | (a) AI IOに統合し「AI連携」1ページ (b) Inbox整理に合流 | (a)。「候補を確認して取り込む」体験がAI Importプレビューと同型のため。Inboxは人間の走り書き用に保つ |
| Activity Log の置き場所 | (a) Todayヘッダーの1ボタン化 (b) AI連携ページへ移設 | (a)を先行、運用で(b)を再判断 |
| ミニ表示ボタン | (a) Todayヘッダーに残す (b) トレイメニューに寄せる | (b)。トレイに同機能があり重複 |

## 4. 実行計画

各フェーズは独立してリリース可能な粒度にし、「作成→保存→再起動→再表示→削除/復元」の縦断検証と
`npm run typecheck` / `build` / `smoke:desktop` / `smoke:model` を完了条件にする。
削除・移設を含むフェーズは、残った導線の実画面検証（到達・作成・保存・再表示）まで行う（`AGENTS.md` Testing）。

### Phase 1: 行き止まりの解消と撤去（小規模・低リスク・即効）

| # | 変更 | 対象 | 補足 |
|---|------|------|------|
| 1-1 | MilestonePage を削除 | `pages/MilestonePage.tsx` | Today「近いマイルストーン」とTimelineが同責務を担う。復活させる場合はTimelineの表示トグルとして再設計 |
| 1-2 | 付箋メモに「Inboxへ送る」を追加 | `MicroMemoPage.tsx`、persistence | kind を null 化して untriaged へ。カード上の1ボタン。これで走り書き→整理→タスク化のループが閉じる |
| 1-3 | 孤立フォームの撤去 | `drawer.tsx`、`WorkspaceApp.saveForm` | source_record / field_definition / task_dependency / reference のフォームと保存分岐を撤去。**エンティティ定義・DB・Snapshot IOは維持**（互換条件） |
| 1-4 | 手動「更新」ボタンの撤去 | `TodayPage.tsx`、`InboxPage.tsx` | 前提検証: capture window / MCP write / 別ウィンドウ更新の全経路で `onWorkspaceChanged` が届くことを実動確認してから撤去。届かない経路があればそちらを直す |
| 1-5 | 命名の統一 | 各ページ・`routes.ts` | 「チャット参照」に統一、「AI Proposal Inbox」→「AI提案の確認」等。ナビ名とページタイトルを一致させる |

### Phase 2: Todayの減量

| # | 変更 | 内容 |
|---|------|------|
| 2-1 | Activity Log ストリップ廃止 | ヘッダーの「活動ログ出力」1ボタンに集約し、押したときだけ日付・出力先を出す。`buildActivityLog` は `lib/` へ移動 |
| 2-2 | ヘッダーを3ボタン以下に | ミニ表示はトレイへ寄せ、更新は1-4で撤去済み。残り: コピー / 活動ログ / タスク追加 |
| 2-3 | セクションの再構成 | 上段=判断(次にやること+今日やること)、下段=リスク(期限切れ・期限が近い待ち・近いマイルストーン)。「Inbox未整理」「予定なし」はメトリクスカード+遷移リンクに格下げし、行リストは出さない（操作はInbox/ToDoで行う） |
| 2-4 | 行アクションの削減 | Todayの行は 完了 / 今日トグル / +1d / 行クリック の4つに絞る。旗と+7dはToDoとドロワーに残す |
| 2-5 | ContextPane の責務整理 | ContextPaneは「選択テーマの文脈」（現在地・待ちの手触り・再発見）に特化し、Todayと重複する期限切れブロックは route=today のとき非表示 |

## 5. やらないこと

- 本整備の中での新機能の追加。機能アイデアは §7 のバックログに分離し、Phase 2 完了以降に個別判断する。Saved View / Custom Field / LogEntry のUI新設は運用で必要性が確認されてから（`docs/PLAN.md` 継続改善の方針どおり）。
- DBスキーマ変更・データの物理削除。孤立フォームの撤去はUI層のみで、既存データとSnapshot互換を維持する。
- design-standard/ 自体の変更（必要が出たら事前相談）。
- ガント（Timeline）の機能改修。本計画は導線整理に限定する。

## 6. 完了条件

1. ナビの全ルート × 主要エンティティで「入口→一覧→詳細→編集削除→永続化→再起動後再表示」が通り、行き止まりゼロ。
2. 到達不能ページ・孤立フォーム・開く導線のない保存分岐がゼロ。
3. Todayのヘッダー操作3個以下、セクション5個以下、1行の操作要素4個以下。
4. ナビ11ルート以下、カウントバッジは行動を促す数のみ。
5. 各フェーズで typecheck / build / smoke:desktop / smoke:model が成功し、パッケージ版での実動smokeを通過。

## 7. 機能アイデア・バックログ（整備後に個別判断）

選定基準は本計画の原則と同じにする。すなわち (a) 既に保存しているデータを活かす、(b) 既存ループを閉じる・深める、(c) ナビ項目を増やさない、(d) ローカルファースト・クリップボードAI往復の方針を守る。
新しい画面・新しいエンティティを最初に置く案は原則不採用とし、既存画面への「1ボタン・1レーン」で始める。

### 7.1 優先候補（ミッション「現在地を失わない」に直結）

### 7.2 二次候補（体験の摩擦を削る小粒）

| # | アイデア | 内容 | 規模 |

### 7.3 採用しないと決めるもの（アンチバックログ）

- カンバンビュー・工数/見積もり管理・ポモドーロ等の一般タスク管理機能。SPECの非目標（重い運用にしない）に反する。
- LLM APIの内蔵呼び出し。クリップボード往復とread-only MCP + Proposal Inboxの方針を維持する（会社環境の制約とローカルファーストの一貫性）。
- Custom Field / Saved View / LogEntry のUI先行実装。§2.1 の「保存だけのデータ」を増やす方向であり、運用上の必要が言語化されてから。
- 通知の常用化（デスクトップ通知の多用）。現在地の確認はTodayと週次レビューへの能動アクセスで担う。
